Design: Data Flow & Core Interfaces

How events flow from user input to screen output

Event Flow (Simplified)

User types prompt in terminal
↓
app.py prompt_async() → check /commands → session.run(prompt)
↓ async event stream
stream_view.py on_event(event) → update Rich.Live display
↓
Terminal Rich renders Markdown + tool status panels

stream_view.py — StreamView
class StreamView:
    def __init__(console)
        self._live = Live(console=console)
        self._text = ""
        self._tools: dict[str, ToolStatus] = {}
        self._working = False

    def start() / stop()
        """Start/stop Rich.Live context."""

    def on_event(event)
        """Dispatch table (simplified):
        agent_start     → _working = True
        text_delta      → _text += delta
        tool_exec_start → _tools[id] = running
        tool_exec_end   → _tools[id] = done
        agent_end       → _working = False
        → _refresh()
        """

    def _refresh()
        """Build renderable and update Live.
        Group([
            Markdown(self._text),
            *tool_panels,
            spinner if working
        ])
        """

Key Design Decisions

No StateManager
State lives directly in StreamView as simple instance vars (_text, _tools, _working). No separate state layer needed for ~3 fields.
No Adaptive Refresh
Call _refresh() directly after each event. Rich.Live handles rate-limiting internally (refresh_per_second=10).
Rich.Markdown for Highlight
Rich.Markdown already does code block highlighting via Pygments. No need for a custom StreamingSyntaxHighlighter.
Scrollback via Console.print
When agent turn completes, stop Live and print final Markdown to console (scrollback). No CommitManager needed.

Extension UI Stub

The existing extensions/ui.py defines a no-op UI protocol. Since we're removing the Extension UI bridge, the no-op implementation stays as-is — extensions just get silent no-ops. No adapter code needed.
